# 06. 와이어프레임 피드백 & 결정 사항 (프론트 개발자 전달용)

> **작성**: 프론트 기획자
> **일자**: 2026-06-18
> **대상**: 프론트 개발자 (`frontend/` 실제 구현 담당)
> **기준 산출물**: `wireframe/` (포트 5174) — GRAVITY 디자인 가이드 적용 완료
> **참조**: `01-information-architecture.md`, `02-wireframes.md`, `work-breakdown/01-api-contract.md`, `02-shared-contracts.md`

이 문서는 와이어프레임 리뷰에서 나온 기획자 피드백과 그에 대한 결정·근거를 정리한 것이다.
와이어프레임(`wireframe/src/screens/*.tsx`)에는 이미 반영되어 있으며, 실제 `frontend/` 구현 시 동일하게 적용한다.
일부 항목은 **백엔드/팀 합의가 필요**하므로 §4를 반드시 확인할 것.

## 1. 전역 (디자인 가이드 적용)

| # | 요청 | 반영 내용 | 상태 |
|---|------|-----------|------|
| G1 | 디자인 가이드 전체 반영 | GRAVITY 토큰(컬러/타이포/Radius/Elevation) 적용. 공통 키트 `wireframe-kit.tsx` 기준 | ✅ 와이어프레임 반영 |
| G2 | 상태 칩 통일 | 모든 상태/유형/긴급도 칩에서 **도트(dot) 제거** → 닷 없는 형태로 통일 | ✅ 반영 |
| G3 | 검색/필터 드롭다운도 디자인 가이드 적용 | 텍스트로만 표기하던 필터를 가이드의 **Select(보더+chevron)·Search(아이콘 인풋)** 스타일로 교체 | ✅ 반영 |

- 컬러: Primary `#05539A`, Accent `#06ADEC`(강조 한정), Neutral Gray, 상태색(success/warning/error)
- 타이포: Pretendard(국문) / DM Sans(영문·숫자)
- 활성 상태는 Primary, Accent는 하이라이트 전용 (토글 활성색을 Accent→Primary로 교체)

## 3. 신규 공통 컴포넌트 (`wireframe-kit.tsx`)

실제 구현에서도 재사용을 권장한다.

| 컴포넌트 | 용도 |
|----------|------|
| `Badge` | 공통 칩 (default/info/success/warning/error/accent), 닷 옵션은 미사용으로 통일 |
| `UrgencyBadge` | 긴급도 (HIGH=error / NORMAL=warning / LOW=default) |
| `TypeBadge` | 문의 유형 칩 (결제/아이템지급/계정/기타 색상 매핑) |
| `StatusBadge` | 문의 상태 칩 (한글 라벨 → 색상 매핑) |
| `Select` | 필터 드롭다운 (보더 + chevron) |
| `SearchBox` | 검색 인풋 (prefix 아이콘) |
| `Pagination` | 페이지네이션 (현재 페이지 Primary) |

## 4. 팀/백엔드 합의 필요 (중요)

> 아래는 와이어프레임에 임시 반영했으나 **확정 전 구현 보류** 또는 **백엔드 협의**가 필요한 항목.

### 4-A. 칸반 "접수·분석중" 표시 그룹핑
- **결정(기획)**: 보드에서 `RECEIVED` + `AI_ANALYZING`을 "접수·분석중" 한 컬럼으로 표시. (둘 다 시스템 자동 처리 구간, 운영자 무액션)
- **제약**: 백엔드 `InquiryStatus` enum 7종은 **변경하지 않는다.** 프론트가 `GET /dashboard/board` 응답의 두 그룹을 합쳐 렌더링.
- **확인 필요**: 백엔드/팀 동의 여부. enum 자체 통합은 하지 않는 것으로 정리.

### 4-B. 유형(Type) → 색상 매핑
- 디자인 가이드 원칙("화면당 색상 3종 이하", "Accent는 강조 한정")과 일부 상충 소지. 유형 식별성을 위해 4색 사용.
- **확인 필요**: 디자이너/기획 최종 승인. 색상 매핑은 `TypeBadge` 한 곳에서 관리.

### 4-C. 완료시각(completedAt) 필드
- 리스트/카드에 완료시각 표시하려면 백엔드 응답에 **발송 완료 시각 필드 추가** 필요.
- 현재 `01-api-contract.md` §4 card DTO에는 `createdAt`만 있음.
- **백엔드 C 요청**: `GET /dashboard/inquiries`, `GET /dashboard/board` card에 `completedAt`(또는 `sentAt`, SENT 시각) 추가. 미완료 건은 `null`.

### 4-D. 고객 문의 폼 — 로그인 컨텍스트 전달 방식
- 유저 ID·닉네임 자동 입력의 전제: 고객이 **게임 로그인 상태**에서 문의 페이지로 진입.
- `POST /api/v1/inquiries` 요청 본문에는 `customerInfo`(userId, nickname, channel)가 이미 있음.
- **확인 필요**: 프론트가 이 값을 어디서 받는가? (게임 클라이언트가 토큰/파라미터로 전달? 별도 세션 조회 API?) — 진입 경로/연동 방식 협의 필요.

### 4-E. 비밀번호 찾기 플로우
- 와이어프레임에 링크만 추가. **플로우·API 미정.**
- 현재 API 계약에 비밀번호 재설정 엔드포인트 없음.
- **확인 필요**: MVP 범위 포함 여부, 포함 시 백엔드 A와 재설정 플로우(이메일? 관리자 초기화?) 및 API 정의 협의.
