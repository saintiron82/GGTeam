# 병렬 작업 계획: 의존성 · 마일스톤 · 병렬 가능 시점

> **핵심 원칙**: 계약(01 API, 02 스키마, 03 인터페이스)을 먼저 확정하면 5명이 서로 기다리지 않고 병렬 작업 가능.

---

## 1. 의존성 그래프

```
[M0: 계약 확정] ──────────────────────────────────┐
   (01 API, 02 스키마/enum, 03 Java 인터페이스)      │ 모두의 전제
        │                                          │
        ├─> 백엔드 A (기반: 엔티티/Repo/인증/상태머신) ──┐
        │        │                                     │
        │        ├─> 백엔드 B (AI 파이프라인) [Mock 선행] │
        │        └─> 백엔드 C (워크플로우/대시보드) [stub 선행]
        │                                              │
        ├─> 프론트 기획 (와이어프레임/UX) ──> 프론트 개발 (MSW 선행)
        │                                              │
        └──────────────> [M3: 통합] <─────────────────┘
```

## 2. 마일스톤

### M0 — 계약 확정 (Day 0, 전원 참여)
- 01 API 계약, 02 공유 스키마, 03 인터페이스 리뷰 & 합의
- **완료 기준**: 5명이 계약 문서에 동의. 이후 병렬 착수.
- 산출: 확정된 계약 문서, 공유 타입 정의(`types.ts`, Java enum/interface)

### M1 — 기반 & 독립 개발 (병렬 시작)
| 담당 | 작업 | 선행 의존 | 병렬 가능 |
|------|------|-----------|-----------|
| 백엔드 A | 엔티티/Repo/인증/상태머신 | M0 | ✅ 즉시 |
| 백엔드 B | LlmClient/MockLlmClient/분석·진단·초안 | M0 (엔티티는 Mock 선행) | ✅ 즉시 (Mock) |
| 백엔드 C | 워크플로우/대시보드 (stub) | M0 (A 인터페이스 stub) | ✅ 즉시 (stub) |
| 프론트 기획 | 와이어프레임/UX/매핑표 | M0 | ✅ 즉시 |
| 프론트 개발 | 공통(ApiClient/Auth)+로그인 | M0 + MSW | ✅ 즉시 (MSW) |

### M2 — 모듈 결합 (백엔드 내부 통합)
- 백엔드 A 엔티티/Repo 완료 → B/C가 Mock/stub을 실제 구현으로 교체
- 백엔드 C ← B의 DraftResponseService/재분석 연동
- 프론트 개발 ← 기획 와이어프레임 화면별 구현 진행
- **완료 기준**: 각 백엔드 모듈 단위 테스트 통과, 프론트 화면 MSW 기반 동작

### M3 — 통합 (Integration)
- 프론트 ↔ 백엔드 실제 API 연결 (MSW 제거)
- 결제 유형 end-to-end 시나리오 검증
- **완료 기준**: 문의접수→AI분석→배정→승인→발송 전체 흐름 동작

### M4 — 테스트 & 안정화
- 통합 테스트, 버그 수정, 커버리지 80% 달성
- Docker Compose 전체 기동 검증

## 3. 병렬성 요약

| 시점 | 동시 진행 가능 작업 수 |
|------|----------------------|
| M0 이후 | 5명 전원 병렬 (계약 기반 독립 작업) |
| M2 | 백엔드 3명 결합 + 프론트 2명 화면 구현 |
| M3 | 통합은 조율 필요 (페어/순차) |

## 4. 병렬 작업을 가능하게 하는 장치
1. **API 계약(01)**: 프론트는 실제 백엔드 없이 MSW로 개발
2. **Java 인터페이스(03)**: 백엔드 B/C는 A 완성 전 stub/mock으로 개발
3. **MockLlmClient**: 백엔드 B는 AWS Bedrock 없이 파이프라인 개발/테스트
4. **모듈 경계 + 브랜치 전략**: 코드 충돌 최소화
5. **공유 타입 정의**: enum/DTO 단일 소스

## 5. 조율이 필요한 지점 (병렬 불가, 협의 필수)
- 계약 변경 (API/스키마/인터페이스) → 전원 합의
- `persistence` 엔티티 수정 → 백엔드 A 경유
- M3 통합 → 프론트-백엔드 페어
- 상태 전이 규칙 변경 → 백엔드 A + 프론트 기획 공유
