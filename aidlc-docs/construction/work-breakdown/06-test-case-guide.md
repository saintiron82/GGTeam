# 역할별 테스트 케이스 정의 가이드

> **목적**: 각 담당자가 자기 영역의 테스트 케이스를 일관된 기준으로 정의·작성하도록 가이드.  
> **목표 커버리지**: 핵심 비즈니스 로직 80% 이상 (NFR-04).  
> **테스트 케이스 정의 방식**: 기획자 주도로 공동 도출 → 파트별 분리 → 재분배 가능한 형태로 정리 (하단 참조).

---

## 공통 원칙
- 테스트 케이스는 **수용 기준(stories.md)** 과 **비즈니스 규칙(business-rules.md BR-xx)** 에서 도출
- 각 테스트는 스토리/규칙 ID로 추적 가능하게 명명 (예: `US-08_타임아웃_3회재시도`)
- 정상(happy path) + 경계 + 예외(failure) 케이스를 모두 포함
- 테스트는 담당 모듈과 같은 브랜치에서 코드와 함께 작성

## 프론트 기획자
- **수용 시나리오 정의**: 각 화면의 사용자 시나리오를 Given-When-Then으로 작성 → 개발자 테스트의 기준 제공
- **UX 검증 체크리스트**: 빈 상태/로딩/에러 표시, 상태별 버튼 규칙, 접근성(a11y)

## 테스트 케이스 공동 정의 및 재분배 (기획자 주도)

> **우선순위**: 자동 생성 "도구" 구축은 후순위. 핵심은 **기획자와 함께 테스트 케이스(시나리오)를 도출하고, 각 파트 작업자 형태에 맞게 분리하여 재분배 가능한 형태로 정리**하는 것이다.

### 프로세스
1. **공동 도출 (기획자 주도)**: 기획자가 수용 시나리오(Given-When-Then)를 중심으로 테스트 케이스 후보를 정리하고, 각 파트 담당자와 함께 검토하여 케이스를 확정한다.
2. **파트별 분리**: 도출된 케이스를 작업자 형태에 맞게 분류한다.
   - 백엔드 A/B/C: 단위/통합 테스트 케이스 (위 표 기준)
   - 프론트 개발: 컴포넌트/UI 테스트 케이스
   - 통합: end-to-end 시나리오
3. **재분배 가능한 형태로 구축**: 케이스를 스토리/규칙 ID로 태깅하고, 담당 파트별로 재할당할 수 있는 공통 포맷(예: 케이스 ID, 전제, 입력, 기대결과, 담당파트)으로 정리한다.

### 케이스 정의 공통 포맷 (예시)
| 케이스 ID | 시나리오 | 입력 | 기대 결과 | 담당 파트 | 근거(US/BR) |
|-----------|----------|------|-----------|-----------|-------------|
| TC-PAY-01 | 결제완료·아이템미지급 문의 | 결제 SUCCESS, 지급 NOT_DELIVERED | aiType=PAYMENT, 진단=지급누락 | 백엔드 B | US-06,11 |
| TC-ASSIGN-01 | 동시 Pull 배정 | 운영자 2명 동시 요청 | 1명만 배정 성공 | 백엔드 C | US-15,BR-12 |

### 보조 (선택, 후순위)
- 입력 도메인 기반 테스트 데이터 생성 보조 스크립트는 여력이 있을 때 백엔드 B/C 로직에 한해 적용 (필수 아님)
