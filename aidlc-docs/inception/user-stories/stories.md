# AI CS Agent - 사용자 스토리

> **MVP (Phase 1) 범위**: 공통 프레임워크 + 결제 유형 end-to-end  
> **Phase 2**: 계정/버그/이벤트/기타 유형 확장, 담당자 자동 추천, 통계

> **핵심 원칙**: 모든 답변은 운영자 승인을 거쳐 발송된다 (AI 단독 발송 없음 / Human-in-the-loop)

---

## 1. 인증 (Auth)

### US-01: 운영자 로그인 [MVP]

**스토리**: 운영자로서, ID/Password로 로그인할 수 있다, 시스템에 안전하게 접근하기 위해.

**수용 기준**:

**시나리오**: 올바른 자격증명으로 로그인
- Given: 등록된 운영자 계정이 존재할 때
- When: 올바른 ID와 Password를 입력하고 로그인 버튼을 클릭하면
- Then: JWT 토큰이 발급되고 대시보드로 이동한다

**시나리오**: 잘못된 자격증명으로 로그인 시도
- Given: 운영자가 로그인 페이지에 있을 때
- When: 잘못된 ID 또는 Password를 입력하면
- Then: "아이디 또는 비밀번호가 올바르지 않습니다" 에러 메시지를 표시한다

**체크리스트**:
- [ ] ID/Password 입력 폼 제공
- [ ] 로그인 성공 시 JWT 토큰 발급
- [ ] 로그인 실패 시 에러 메시지 표시
- [ ] 비밀번호 5회 실패 시 계정 잠금

---

### US-02: 인증 토큰 관리 [MVP]

**스토리**: 운영자로서, 로그인 상태가 유지될 수 있다, 매번 재로그인하지 않기 위해.

**수용 기준**:

**시나리오**: 토큰 만료
- Given: 운영자가 로그인된 상태에서
- When: JWT 토큰이 만료되면
- Then: 로그인 페이지로 리다이렉트된다

**체크리스트**:
- [ ] JWT 토큰 유효시간 설정 (예: 8시간)
- [ ] 인증되지 않은 API 요청 시 401 반환
- [ ] 로그아웃 시 토큰 무효화

---

## 2. 문의 접수 (FR-01)

### US-03: 고객 문의 웹 폼 [MVP]

**스토리**: 고객으로서, 웹 폼을 통해 문의를 접수할 수 있다, CS 지원을 받기 위해.

**수용 기준**:

**시나리오**: 문의 접수 성공
- Given: 고객이 문의 접수 페이지에 접근했을 때
- When: 문의 유형을 선택하고 내용을 입력한 후 제출하면
- Then: 고유 문의 ID가 생성되고 접수 완료 메시지를 표시한다

**체크리스트**:
- [ ] 문의 유형 선택 드롭다운 (결제, 계정, 버그, 이벤트, 기타)
- [ ] 문의 내용 텍스트 입력 필드 (필수, 최소 10자)
- [ ] 제출 버튼
- [ ] 접수 완료 시 문의 ID 표시

---

### US-04: 문의 ID 및 타임스탬프 자동 생성 [MVP]

**스토리**: 시스템으로서, 문의 접수 시 고유 ID와 타임스탬프를 자동 생성할 수 있다, 문의를 추적하기 위해.

**수용 기준**:

**시나리오**: 자동 메타데이터 생성
- Given: 고객이 문의를 제출할 때
- When: 서버에서 문의를 저장하면
- Then: UUID 기반 고유 문의 ID와 KST 타임스탬프가 자동 부여된다

**체크리스트**:
- [ ] UUID 형식의 고유 문의 ID 생성
- [ ] 접수 시각 자동 기록 (ISO 8601, KST)
- [ ] ID 중복 불가 보장

---

### US-05: 문의 유형 선택 [MVP]

**스토리**: 고객으로서, 문의 유형을 선택할 수 있다, 적절한 처리 경로로 배정되기 위해.

**수용 기준**:

**시나리오**: 유형 선택
- Given: 문의 접수 폼에서
- When: 유형 드롭다운을 클릭하면
- Then: 결제, 계정, 버그, 이벤트, 기타 옵션이 표시된다

**체크리스트**:
- [ ] 5개 유형 옵션 제공
- [ ] 유형 미선택 시 제출 불가
- [ ] 선택된 고객 유형이 문의 데이터에 저장 (customer_type)

> **참고**: 고객이 선택한 유형(customer_type)은 참고용이며, 실제 처리 유형은 US-06의 AI 분석 결과(ai_type)를 우선한다.

---

## 3. AI 문의 분석 (FR-02)

### US-06: AI 자동 분류 [MVP]

**스토리**: 시스템으로서, 접수된 문의를 AI로 자동 분류할 수 있다, 빠른 처리를 위해.

> **의견 반영**: 고객이 선택한 유형(customer_type)과 AI가 분석한 유형(ai_type)은 다를 수 있으며, AI는 더 세분화된 서브카테고리를 부여한다.

**수용 기준**:

**시나리오**: 결제 문의 자동 분류 (고객 선택과 AI 분석 차이)
- Given: 고객이 "기타"로 선택했으나 "결제 후 아이템 미지급" 내용의 문의가 접수되었을 때
- When: AI 분석이 실행되면
- Then: ai_type=결제, sub_category=아이템미지급, 긴급도=긴급, 핵심키워드=["결제완료","아이템 미지급"]가 반환된다

**체크리스트**:
- [ ] AWS Bedrock API 호출
- [ ] 대분류(ai_type): 결제, 계정, 버그, 이벤트, 기타
- [ ] 세부분류(sub_category): 대분류 하위 상세 유형
- [ ] customer_type과 ai_type을 별도 필드로 저장
- [ ] 긴급도 분류 (긴급/보통/낮음)
- [ ] 핵심 키워드 추출
- [ ] 핵심 내용 요약 생성

---

### US-07: AI 분석 결과 저장 [MVP]

**스토리**: 시스템으로서, AI 분석 결과를 저장할 수 있다, 운영자가 확인하기 위해.

**수용 기준**:

**시나리오**: 분석 결과 저장
- Given: AI 분석이 완료되었을 때
- When: 결과를 DB에 저장하면
- Then: 문의 ID에 연결된 분석 결과(유형, 서브카테고리, 긴급도, 요약, 키워드)가 조회 가능하다

**체크리스트**:
- [ ] 분석 결과를 문의 ID와 연결하여 저장
- [ ] 분석 완료 시각 기록
- [ ] 분석 실패 시 에러 로깅 및 재시도 (US-08 참조)

---

### US-08: AI 분석 실패 처리 [MVP]

**스토리**: 시스템으로서, AI 분석 실패 시 적절히 대응할 수 있다, 문의가 누락되지 않기 위해.

> **의견 반영**: 타임아웃과 API 에러(인증/요청 오류 등)는 원인이 다르므로 구분하여 처리한다.

**수용 기준**:

**시나리오**: Bedrock 호출 타임아웃
- Given: AI 분석 요청이 전송되었을 때
- When: 응답이 타임아웃(120초)을 초과하면
- Then: exponential backoff로 최대 3회 재시도 후, 실패 시 "수동분류대기" 상태로 전환한다

**시나리오**: Bedrock API 에러 (4xx/5xx)
- Given: AI 분석 요청이 전송되었을 때
- When: 인증 오류나 요청 오류가 반환되면
- Then: 재시도하지 않고 즉시 "수동분류대기"로 전환하고 에러 유형을 로깅한다

**체크리스트**:
- [ ] API 타임아웃 설정 (120초, NFR-01 600초 범위 내 조정 가능)
- [ ] 타임아웃 시: 최대 3회 재시도 (exponential backoff)
- [ ] API 에러(4xx/5xx) 시: 재시도 없이 즉시 실패 처리
- [ ] 에러 유형을 구분하여 로깅 (timeout / api_error)
- [ ] 최종 실패 시 상태를 "수동분류대기"로 변경

---

## 4. 시스템 데이터 조회 (FR-03)

### US-09: 결제 이력 조회 [MVP]

**스토리**: 시스템으로서, 결제 관련 문의 시 결제 이력을 조회할 수 있다, 원인 분석 데이터를 확보하기 위해.

**수용 기준**:

**시나리오**: 결제 실패 문의 데이터 조회
- Given: 결제 유형으로 분류된 문의가 있을 때
- When: 시스템 데이터 조회가 트리거되면
- Then: 해당 고객의 최근 결제이력, 성공/실패 여부, 오류로그가 반환된다

**체크리스트**:
- [ ] 고객 식별 정보로 결제이력 테이블 조회
- [ ] 결제 성공/실패 상태 포함
- [ ] 오류 코드 및 로그 포함
- [ ] 최근 N건 제한 (예: 최근 10건)

---

### US-10: 데이터 조회 확장 구조 [Phase 2]

**스토리**: 시스템으로서, 문의 유형별로 다른 DB 테이블을 조회할 수 있다, 각 유형에 맞는 데이터를 확보하기 위해.

**수용 기준**:

**시나리오**: 계정 문의 데이터 조회
- Given: 계정 유형으로 분류된 문의가 있을 때
- When: 시스템 데이터 조회가 트리거되면
- Then: 해당 고객의 계정 상태, 로그인 이력, 보안 이벤트가 반환된다

**체크리스트**:
- [ ] 유형별 조회 전략 패턴 (Strategy Pattern)
- [ ] 계정: 계정 상태, 로그인 이력
- [ ] 버그: 에러 로그, 시스템 상태
- [ ] 이벤트: 참여 이력, 보상 내역

> **참고**: MVP에서는 US-09(결제)만 구현하되, 이 확장 구조(Strategy Pattern)는 MVP 설계 시 반영한다.

## 5. 상태 진단 및 처리 방향 제안 (FR-04)

### US-11: AI 원인 진단 [MVP]

**스토리**: 시스템으로서, 조회된 데이터를 기반으로 원인을 진단할 수 있다, 처리 방향을 결정하기 위해.

> **의견 반영**: AI 진단 결과만으로 고객에게 답변이 발송되는 경우는 없다. 반드시 운영자 검토를 거친다.

**수용 기준**:

**시나리오**: 결제 실패 원인 진단
- Given: 결제 이력에서 "시스템 오류로 아이템 미지급"이 확인되었을 때
- When: AI 진단이 실행되면
- Then: 원인="시스템 오류로 아이템 지급 누락", 처리방향="보상 지급 및 안내" 결과를 생성한다

**체크리스트**:
- [ ] 조회 결과를 AI에 입력하여 원인 분석
- [ ] 진단 결과: 원인, 처리 방향, 신뢰도 포함
- [ ] AI 진단 결과는 반드시 운영자 검토를 거침 (자동 발송 없음)

---

### US-12: 담당자 알림 발송 [MVP]

**스토리**: 시스템으로서, 처리가 필요한 문의를 운영자가 인지할 수 있도록 알림을 표시할 수 있다, 빠른 대응을 위해.

> **의견 반영**: 티켓 배정은 운영자가 미배정 목록에서 직접 가져가는 Pull 방식 (소규모팀 운영, US-15 참조). 본 스토리는 미배정/긴급 건에 대한 알림 표시에 집중한다.

**수용 기준**:

**시나리오**: 미배정 문의 알림
- Given: 문의가 "담당자배정대기" 상태일 때
- When: 운영자가 대시보드에 접속하면
- Then: 미배정 문의 건수와 긴급 건이 알림 영역에 표시된다

**체크리스트**:
- [ ] 대시보드 내 미배정 문의 카운트 표시
- [ ] 긴급 문의 강조 표시
- [ ] 알림에 문의 ID, 긴급도, 요약 포함

---

## 6. 답변 초안 생성 (FR-05)

> **결정 반영**: 답변 초안은 AI 분석 파이프라인에서 진단 직후 **미리 자동 생성**된다. 운영자는 대시보드에서 이미 준비된 초안을 검토한다.

### US-13: AI 답변 초안 자동 생성 [MVP]

**스토리**: 시스템으로서, 진단 완료 직후 AI가 고객 응대 답변 초안을 자동 생성할 수 있다, 운영자가 즉시 검토할 수 있도록 하기 위해.

**수용 기준**:

**시나리오**: 진단 후 답변 자동 생성
- Given: US-11 원인 진단이 완료되었을 때
- When: 답변 초안 생성이 자동으로 이어서 실행되면
- Then: 고객 친화적인 안내 문구가 생성되어 문의에 저장되고 "담당자배정대기" 상태가 된다

**체크리스트**:
- [ ] 진단 결과 기반 맞춤 답변 자동 생성
- [ ] 고객 친화적 어조 사용
- [ ] 답변 초안을 DB에 저장
- [ ] 운영자가 대시보드 진입 시 초안이 이미 준비된 상태

---

### US-14: 답변 초안 수정 [MVP]

**스토리**: 운영자로서, AI가 생성한 답변 초안을 수정할 수 있다, 정확하고 적절한 답변을 보내기 위해.

**수용 기준**:

**시나리오**: 답변 수정
- Given: AI 답변 초안이 표시된 상태에서
- When: 운영자가 내용을 수정하고 저장하면
- Then: 수정된 답변이 저장되고 수정 이력이 기록된다

**체크리스트**:
- [ ] 텍스트 에디터로 답변 편집 가능
- [ ] 수정 전/후 내용 이력 보관
- [ ] 저장 버튼으로 수정 완료

---

## 7. 담당자 배정 및 승인 워크플로우 (FR-06, FR-07)

### US-15: 문의 직접 배정 (Pull) [MVP]

**스토리**: 운영자로서, 미배정 문의 목록에서 문의를 직접 가져올 수 있다, 스스로 처리할 건을 선택하기 위해.

> **결정 반영**: 소규모팀(1~5명)이므로 팀장 일괄 배정 없이 운영자가 직접 가져가는 Pull 방식 채택.

**수용 기준**:

**시나리오**: 문의 가져오기
- Given: "담당자배정대기" 상태의 미배정 문의가 있을 때
- When: 운영자가 해당 문의의 "담당하기" 버튼을 클릭하면
- Then: 문의의 담당자가 본인으로 지정되고 상태가 "운영자확인중"으로 전환된다

**체크리스트**:
- [ ] 미배정 문의 목록 제공
- [ ] "담당하기" 버튼으로 본인 배정
- [ ] 배정 시 담당자 ID 및 시각 기록
- [ ] 이미 배정된 문의는 중복 배정 방지 (동시성 처리)
- [ ] 필요 시 담당자 변경 가능

---

### US-16: 답변 승인 [MVP]

**스토리**: 운영자로서, 답변을 승인할 수 있다, 고객에게 답변을 발송하기 위해.

**수용 기준**:

**시나리오**: 승인 처리
- Given: 답변 초안(또는 수정본)이 준비된 상태에서
- When: 운영자가 "승인" 버튼을 클릭하면
- Then: 상태가 "승인완료"로 변경되고 답변 발송이 트리거된다

**체크리스트**:
- [ ] 승인 버튼 제공
- [ ] 승인 시 상태 변경 (운영자확인중→승인완료)
- [ ] 승인자 ID 및 시각 기록
- [ ] 답변 발송 트리거

### US-17: 답변 반려 및 AI 재생성 [MVP]

**스토리**: 운영자로서, 부적절한 답변을 반려 사유와 함께 반려할 수 있다, AI가 사유를 반영해 답변을 재생성하도록 하기 위해.

> **결정 반영**: 반려 사유는 단순 기록이 아니라, AI가 입력으로 받아 재생성 시 반영한다.

**수용 기준**:

**시나리오**: 반려 후 AI 재생성
- Given: 답변 초안이 부적절할 때
- When: 운영자가 반려 사유를 입력하고 "반려 및 재생성" 버튼을 클릭하면
- Then: 반려 사유가 AI에 전달되어 사유를 반영한 새 답변 초안이 생성된다

**체크리스트**:
- [ ] 반려 버튼 및 사유 입력 필드 (필수)
- [ ] 반려 사유를 AI 재생성 프롬프트에 포함
- [ ] 재생성 요청 사유 및 시각을 이력에 기록
- [ ] 재생성 횟수 추적
- [ ] 재생성된 초안을 운영자가 다시 검토

---

### US-18: 승인/반려 이력 조회 [MVP]

**스토리**: 운영자로서, 승인/반려 이력을 조회할 수 있다, 처리 과정을 추적하기 위해.

**시나리오**: 이력 조회

- When: 승인 이력 탭을 클릭하면
- Then: 승인/반려/재생성 일시, 담당자, 사유가 시간순으로 표시된다

**체크리스트**:
- [ ] 승인/반려/수정/재생성 이벤트 모두 기록
- [ ] 시간순 정렬
- [ ] 담당자 정보 포함

## 8. 처리 현황 관리 (FR-07)

### US-19: 문의 상태 자동 전이 [MVP]

**스토리**: 시스템으로서, 각 처리 단계에서 상태를 자동 전이할 수 있다, 정확한 현황 파악을 위해.

**수용 기준**:

**시나리오**: 상태 전이 흐름
- Given: 문의가 "접수" 상태일 때
- When: AI 분석이 시작되면
- Then: 상태가 "AI분석중"으로 변경된다

**체크리스트**:
- [ ] 상태 흐름: 접수→AI분석중→담당자배정대기→운영자확인중→승인완료→발송완료
- [ ] 각 전이 시 타임스탬프 기록
- [ ] 비정상 전이 방지 (상태 머신)

---

### US-20: 칸반 보드형 처리 현황 UI [MVP]

**스토리**: 운영자로서, 문의를 상태별 칸반 보드로 볼 수 있다, 전체 처리 흐름을 한눈에 파악하기 위해.

> **결정 반영**: Jira/Mantis 스타일의 티켓 보드형(칸반) UI 채택. 상태별 컬럼에 문의 카드를 배치한다.

**수용 기준**:

**시나리오**: 칸반 보드 표시
- Given: 운영자가 대시보드에 접속했을 때
- When: 보드 뷰가 로드되면
- Then: 상태별 컬럼(접수/AI분석중/담당자배정대기/운영자확인중/승인완료/발송완료)에 문의 카드가 배치되어 표시된다

**체크리스트**:
- [ ] 상태별 컬럼 레이아웃 (칸반)
- [ ] 각 카드에 문의 ID, 유형, 긴급도, 요약 표시
- [ ] 긴급 카드 시각적 강조
- [ ] 카드 클릭 시 상세 페이지 이동
- [ ] (선택) 리스트 뷰 토글 제공
